DBMS (Database Management System)

Architecture overview

A query goes through a chain of components in order to retrieve or modify data :

‚ú≥Ô∏è Query processor: parses, interprets, and validates the query.

‚ú≥Ô∏è Query optimizer:

  • First, it eliminates impossible and redundant parts of the query.
  • Then attempts to find the most efficient way to execute the query based on internal statistics (index cardinality, approximate intersection size, etc.) and data placement (which nodes in the cluster hold the data and the costs associated with its transfer).

‚ú≥Ô∏è Transaction manager: schedules transactions and ensures they cannot leave the database in a logically inconsistent state.

‚ú≥Ô∏è Lock manager: locks database objects for running transactions, ensuring that concurrent operations do not violate the physical integrity of the data.

‚ú≥Ô∏è Together, transaction and lock managers are responsible for concurrency control: they guarantee the logical and physical integrity of data while ensuring that concurrent operations are performed as efficiently as possible.

‚ú≥Ô∏è Access methods: manage access and organizing data on disk. Access methods include heap files and storage structures such as B-Trees or LSM Trees.

‚ú≥Ô∏è Buffer manager: caches data pages in memory.

‚ú≥Ô∏è Recovery manager: maintains the operation log and restoring the system state in case of a failure.

Memory vs Disk DBMS

According to the storage technique, we can find two types of DBMS:

  • In-memory DBMSs store data primarily in memory and use the disk for recovery and logging.
  • Disk-based DBMSs hold most of the data on disk and use memory for caching disk contents or as a temporary storage.

Databases using memory as their primary data store do so primarily because of performance (low access costs) and also because programming for memory (RAM) is much easier than for disk.

But, there are also a lot of limitations to consider for in-memory mode:

  • RAM contents are not persistent: software errors, crashes, hardware failures, and power outages can result in data loss.
  • There are ways to ensure durability, such as uninterrupted power supplies and battery-backed RAM, but they require additional hardware resources and operational expertise.
  • Disks are also easier to maintain and have significantly lower prices.

Aha time!

What I take from this comparison: the perfect software does not exist, it‚Äôs just a trade-off:

  • If you are looking for speed and simplicity, In-memory DBMS will be the appropriate solution (for example, saving the most searched results in the Redis cache).
  • However, if the data is sensitive and requires persistence (such as user accounts and their contact details), you should opt for Disk-based DBMS.
  • At the same time, nothing prevents to mix the two concepts to have a better performance result.

There are other features that differentiate DBMSs, not only the storage mode, such as how to organize data in rows or columns, let‚Äôs take a look!

Column Vs Row Oriented DBMS

‚ú≥Ô∏è Row-oriented database management systems store data in records or rows. Their layout is quite close to the tabular data representation, where every row has the same set of fields.

‚ú≥Ô∏è Column-oriented database management systems partition data vertically (i.e., by column) instead of storing it in rows. Values for the same column are stored contiguously on disk.

Let‚Äôs analyze the trade-offs now:

üîµ Row-oriented pros:

  • This approach works well for cases where several fields constitute the record (for example a single user record).
  • All fields representing a single record are often read together.
  • When creating records (for example, when the user fills out a registration form), we write them together as well.
  • At the same time, each field can be modified individually.

‚ö´ Row-oriented cons: queries accessing individual fields of multiple records (for example, queries fetching only the phone numbers) are more expensive.

Let‚Äôs look at the organization by columns now:

üîµ Column-oriented pros:

  • Storing the values of different columns in separate files or file segments is efficient for per-column queries because they can be read in a single pass rather than consuming entire rows and discarding data from columns that don‚Äôt have not been queried.
  • Storing values that have the same data type together (e.g., numbers with other numbers, strings with other strings) offers a better compression ratio.

‚ö´ Column-oriented cons: I think the downside is clear, access to all information about a record is expensive.

Like I said it‚Äôs all about trade-off!

Clustered and non-clustered index

Indexing is the first thing that comes to mind when we consider a database‚Äôs performance.

It‚Äôs like the table of contents in a book: if you need to go to a particular chapter, you go to the table of contents, find the page number of the chapter and go directly to that page:

Without indexes (table of contents), the process of finding the desired chapter would have been very slow !

There are two magic types of indexes:

  • clustered
  • non-clustered

A clustered index defines the order in which data is physically stored in a table. Table data can be sorted in only way, therefore, there can be only one clustered index per table.

The important word in this explanation is: physically.

In SQL Server, the primary key constraint automatically creates a clustered index on that particular column.

It‚Äôs great, isn‚Äôt it? What can we say now about non-clustered index?

‚ú≥Ô∏è A non-clustered index doesn‚Äôt sort the physical data inside the table.

‚ú≥Ô∏è The clustered Index focuses on physical structure. The non-clustered Index focuses on logical structure.

‚ú≥Ô∏è clustered indexes contain data, whereas non-clustered indexes are stored in a different table (which requires more storage space).

‚ú≥Ô∏è There can be only one clustered index per table. However, we can create multiple non-clustered indexes on a single table.

‚ú≥Ô∏è clustered indexes are faster than non-clustered indexes since they don‚Äôt involve any extra lookup step.

‚ú≥Ô∏è clustered index are typically larger because it includes all the data rows. Whereas, non-clustered index only includes index key values and pointers to the data rows.

‚ú≥Ô∏è clustered index is faster for retrieving large ranges of sequential data. However, non-clustered index is faster for retrieving small sets of data or for sorting and aggregating data.

Data Files and Index Files

Database systems use special files for storing the data. They don‚Äôt rely on filesystem hierarchies of directories and files for locating records. They use an implementation-specific formats.

Why? well this for:

  • Storage efficiency: files are organized in a way that minimizes storage overhead per stored data record.
  • Access efficiency: records can be located in the smallest possible number of steps.
  • Update efficiency: record updates are performed in a way that minimizes the number of changes on disk.

To efficiently locate a record, database systems use indexes.

Usually data files and index files are separated and index files are typically smaller than the data files.

Well, but with which data structure, data and indexes are implemented in order to ensure high performance? It‚Äôs a good question!

For example, the B-Tree is the data structure SQLite uses to represent both tables and indexes. Similarly Postgres, MySQL, and Oracle Database.

B-Tree Variants

  • Copy-on-write B-Trees are structured like B-Trees, but their nodes are immutable and are not updated in place. Instead, pages are copied, updated, and written to new locations.
  • Lazy B-Trees reduce the number of I/O requests from subsequent same-node writes by buffering updates to nodes.
  • Bw-Trees separate B-Tree nodes into several smaller parts that are written in an append-only manner. This reduces costs of small writes by batching updates to the different nodes together.
  • Cache-oblivious B-Trees allow treating on-disk data structures in a way that is very similar to how we build in-memory ones.

Log-structured merge-tree (LSM)

Overview

Let‚Äôs look at this beautiful and comprehensive explanation:

The log-structured mergetree (LSM-tree) is a disk-based data structure designed to provide low-cost indexing for a file experiencing a high rate of record inserts (and deletes) over an extended period. The LSM-tree uses an algorithm that defers and batches index changes, cascading the changes from a memory-based component through one or more disk components in an efficient manner reminiscent of merge sort. ‚Äî The log-structured merge-tree (LSM-tree) | SpringerLink

üî∫Due to the growing popularity of write-intensive workloads, the Log-Structured Merge (LSM) Tree is widely used in modern database systems, such as Casandra, AsterixDB, LevelDB, and RocksDB.

üî∫To achieve high write throughput, the LSM-tree buffers writes into components in memory and flushes the memory components to disk via sequential I/O when it is full.

LSM-Tree Data Structure

The main components are Memtable, WAL (Write Ahead Log), SSTable:

‚ú≥Ô∏è Memtable: this data structure is the first point of contact for every read or write operation received by the LSM Tree. The Memtable is an in-memory structure that buffers incoming data from clients before flushing them to disk.

‚ú≥Ô∏è SSTable: the Sorted String table (SSTable), a concept borrowed from Google‚Äôs BigTable, is the on-disk format (a file) where key value entries from Memtable are flushed in sorted order. An SSTable file is created when the Memtable reaches its capacity or when compaction happens (more on that later).

‚ú≥Ô∏è B-Tree enables faster reads and LSM-Tree enables fast writes.
